# Spec : Admin UI v2 — Onglets Trust, Coûts, Workflows, Agents

**Date :** 2026-03-24
**Statut :** Implémenté ✅
**Projet :** nanobot-stack v10
**Scope :** 4 nouveaux onglets dans l'Admin UI existante (Trust Policies, Cost Dashboard, Procedural Workflows, Agent Status)

---

## 1. Contexte & Objectifs

Les features v10 (trust engine, budget tokens, mémoire procédurale, agents) sont implémentées mais leur interface d'administration est absente. Les 4 nouveaux onglets permettent de configurer et monitorer ces systèmes sans accès direct à la base de données.

**Architecture existante :** `admin_ui.py` génère du HTML statique avec Alpine.js pour l'interactivité. Les onglets sont ajoutés via des constantes `SECTION_*` et du JavaScript `*Section()`.

---

## 2. Architecture

### Fichiers modifiés

| Fichier | Modification |
|---------|-------------|
| `src/bridge/admin_ui.py` | Ajout de 4 constantes `SECTION_*`, 4 fonctions Alpine.js |

### Pattern Alpine.js (identique aux onglets existants)

```javascript
function trustSection() {
    return {
        policies: [],
        audit: [],
        async init() { await this.loadPolicies(); },
        async loadPolicies() {
            const r = await fetch('/trust/policies', {headers: AUTH_HEADERS});
            this.policies = await r.json();
        },
        // ...
    }
}
```

## 3. Onglet Trust Policies

**Données sources :** `GET /trust/policies`, `GET /trust/audit`

### Vue liste des policies

Tableau avec colonnes :
- Type d'action (`action_type`)
- Niveau de confiance — dropdown : `blocked` | `approval_required` | `notify_then_execute` | `auto`
- Exécutions réussies / échouées
- Auto-promotion configurée (N exécutions)
- Dernière promotion
- Bouton "Promouvoir" (→ monte d'un niveau, `POST /trust/promote/{type}`)

### Vue audit

Tableau paginé des 50 dernières décisions :
- Timestamp
- Type d'action + détail (tronqué)
- Niveau appliqué
- Outcome : badge coloré (executed ✅ / pending ⏳ / cancelled ❌ / blocked 🚫)
- Bouton "Annuler" si outcome = `pending` (`POST /trust/cancel/{audit_id}`)

## 4. Onglet Cost Dashboard

**Données sources :** `GET /budget/status`, `GET /budget/history`, `GET /budget/usage`

### Indicateurs en en-tête

- Tokens utilisés aujourd'hui / budget quotidien (barre de progression colorée : vert/orange/rouge)
- Coût estimé aujourd'hui vs budget ($)
- Pression budget : `{pressure}%` avec badge coloré

### Graphique 7 jours (Chart.js)

Bar chart : tokens utilisés par jour (7 derniers jours), avec ligne de budget quotidien.

```html
<canvas id="cost-chart"></canvas>
<script>
new Chart(document.getElementById('cost-chart'), {
    type: 'bar',
    data: { labels: [...dates], datasets: [{label: 'Tokens', data: [...usage]}] }
});
</script>
```

### Breakdown par modèle (aujourd'hui)

Tableau : modèle | tokens input | tokens output | coût estimé | % du total

### Projection mensuelle

```
Projection mensuelle : {daily_avg * 30} tokens ≈ ${monthly_cost_estimate}
Budget mensuel : ${monthly_budget}
```

### Alerte budget

Si `budget_pressure >= 0.8` : banner rouge "⚠️ Budget critique — downgrade Ollama actif"

## 5. Onglet Procedural Workflows

**Données sources :** `GET /procedural/workflows`, `GET /procedural/status`
(Endpoints à ajouter dans `extensions.py`)

### Vue liste des workflows appris

Tableau avec colonnes :
- Pattern déclencheur
- Nombre d'étapes
- Fréquence observée (Nx)
- Confiance (barre de progression)
- Dernière observation (date relative)
- Auto-suggest : toggle
- Actions : Détail | Exécuter maintenant | Supprimer

### Vue détail (panneau latéral)

- Pattern déclencheur
- Séquence d'étapes numérotées (action + params condensés)
- Historique d'exécution (succès/échec)

### Indicateurs globaux

- Workflows appris : N
- Exécutions replay ce mois : N
- Tokens économisés (estimation) : N

---

## 6. Onglet Agent Status

**Données sources :** `GET /agent/status`, `GET /agent/history`

### Agents disponibles

Carte par agent :
- Nom + description
- Outils disponibles (liste)
- Statut : actif / désactivé (toggle relié à env var)

### Exécutions récentes

Tableau des 20 dernières exécutions via `/agent/run` :
- Timestamp
- Tâche (tronquée 100 chars)
- Agents impliqués
- Durée (ms)
- Tokens utilisés
- Statut : completed ✅ / needs_approval ⏳ / failed ❌

### Métriques globales

- Exécutions totales ce mois : N
- Tokens économisés par délégation (estimation)
- Taux de succès : N%

## 7. Endpoints API supplémentaires nécessaires

Ces endpoints doivent être ajoutés dans `extensions.py` pour alimenter les nouveaux onglets :

| Méthode | Endpoint | Module source | Description |
|---------|----------|---------------|-------------|
| GET | `/procedural/workflows` | `procedural_memory.py` | Liste des workflows appris |
| GET | `/procedural/status` | `procedural_memory.py` | Compteurs globaux |
| POST | `/procedural/workflows/{id}/toggle` | `procedural_memory.py` | Toggle `auto_suggest` |
| DELETE | `/procedural/workflows/{id}` | `procedural_memory.py` | Supprimer un workflow |

---

## 8. Ordre d'implémentation

1. Endpoints manquants dans `extensions.py` (procedural/workflows)
2. `SECTION_TRUST` + `trustSection()` dans `admin_ui.py`
3. `SECTION_COST` + `costSection()` + intégration Chart.js
4. `SECTION_WORKFLOWS` + `workflowsSection()`
5. `SECTION_AGENTS` + `agentsSection()`
6. Tests : rendu HTML, appels API corrects, toggles fonctionnels

---

## 9. Contraintes techniques

- Chart.js chargé depuis CDN dans le HTML existant (déjà inclus pour l'onglet Scheduler)
- `AUTH_HEADERS` déjà défini dans le JS global de l'Admin UI existante
- Les 4 onglets suivent exactement le même pattern HTML/Alpine.js que les onglets existants (pas de nouvelle dépendance)
- Les données sont chargées lazily au clic sur l'onglet (`x-show` + `@click` Alpine.js)
